# 上下文精炼与关联规则

## 基本信息

| 字段 | 内容 |
|------|------|
| 规则名称 | 上下文精炼与关联规则 |
| 规则目标 | 把原始产品沟通转化为结构化、可追溯、可关联的产品上下文 |
| 触发页面 | 01-sources/（采集完成后触发） |

## 前置条件

1. 01-sources/ 目录存在且包含至少一份原始材料
2. 01-sources/check.md 已生成且通过

## 执行步骤

### 精炼步骤

1. 读取 01-sources/ 下所有原始材料
2. 提取需求事实，写入 facts.md
3. 提取设计决策，写入 decisions.md
4. 提取业务约束，写入 constraints.md
5. 标记冲突点，写入 conflicts.md
6. 标记待确认问题，写入 questions.md
7. 记录 AI 推断，写入 assumptions.md
8. 运行精炼检查，写入 check.md

### 关联步骤

1. 读取 02-refined/ 下所有精炼结果
2. 建立需求→页面关联，写入 page-map.md
3. 建立功能→规则关联，写入 feature-map.md
4. 建立规则→数据关联，写入 rule-map.md
5. 建立数据对象关联，写入 data-map.md
6. 建立验收关联，写入 acceptance-map.md
7. 建立上下文关联，写入 context-map.md
8. 运行关联检查，写入 check.md

## 状态变化

| 状态 | 触发条件 | 变化结果 |
|------|---------|---------|
| 待精炼 | 采集完成 | 开始精炼 |
| 精炼中 | 开始提取 | 生成 02-refined/ 文件 |
| 精炼完成 | check.md 通过 | 可以进入关联 |
| 待关联 | 精炼完成 | 开始关联 |
| 关联中 | 开始建立关系 | 生成 03-relations/ 文件 |
| 关联完成 | check.md 通过 | 可以进入生成 |

## 异常分支

| 异常场景 | 处理方式 |
|---------|---------|
| 原始材料来源不明 | 在 check.md 中标记，要求补充 |
| 事实与推断无法区分 | 默认标记为推断，待人工确认 |
| 存在冲突未裁决 | 在 conflicts.md 中记录，保持待处理状态 |
| 存在孤立需求 | 在 check.md 中标记，要求补充关联 |

## 权限规则

| 角色 | 权限 |
|------|------|
| 产品经理 | 可确认事实、决策、约束、冲突、问题 |
| Agent | 可提取和标记，不可确认 |
| 研发 | 可查看，可补充技术约束 |

## 幂等要求

重复执行精炼和关联时：
- 已存在的事实不重复创建
- 已存在的关联不重复建立
- 新增材料只处理新增部分

## 日志审计

需要记录：
- 每次精炼处理了哪些材料
- 每次新增了哪些事实、决策、约束
- 每次建立了哪些关联
- 每次检查发现了哪些问题

## 关联数据对象

- source（原始材料）
- fact（需求事实）
- decision（设计决策）
- constraint（业务约束）
- conflict（冲突点）
- question（待确认问题）
- assumption（AI推断）
- relation（关联关系）

## 建议接口用途

本规则为 Skill 内部规则，不涉及外部接口。Agent 通过读取和写入 Markdown 文件执行。
