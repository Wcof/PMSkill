# ADR 0009: pm-refine 双执行模式——追问模式为默认

## Status

Accepted (2026-07-01). Grilling 会话 6 轮完成，所有待定项已锁定。

## Context

pm-refine 当前反例黑名单明文禁止"逐个确认"（第 225 行：`需要 PM 逐个确认才能继续 → 反模式`），设计哲学是"推断优先于追问"（CONTEXT.md 旧版）。这假设 PM 已想清楚，Agent 只需快速沉淀。

但实际场景中 PM 常自己也模糊——需要被引导问对问题。参考 `/Users/ldh/Downloads/project/skills` 中的 grill-with-docs（grilling + domain-modeling 组合），其价值在于"relentless interview to sharpen a plan"——通过逐个追问把模糊想法压成清晰设计。

矛盾点：pm-refine 的"零确认"契约与"追问讨论"需求互斥。

## Decision

**pm-refine 引入双执行模式，追问模式为默认**：

| 模式 | 触发 | 行为 | PM 介入 |
|---|---|---|---|
| 追问模式（默认） | `pm-need <需求>` 正常模式 / `pm-refine` 独立调用 | 8 维逐维推断，推断不了的项转为向 PM 提问，每问附推荐答案 | 高（逐维回答） |
| 自主推断模式 | `pm-need <需求> --auto` / `pm-refine --auto` | 8 维自主推断，推断不了的标 `[待确认]` 记入信息缺口；内部完成自我追问 loop 但不外显 | 零 |

**关键约束**：两种模式**共享 8 维精炼骨架**——精炼是单一领域动作，执行模式是正交维度。不拆成 pm-grill + pm-refine 两个 skill。

**反转现有契约**：
- pm-need 正常模式从"能扫到绝不问"变为"正常模式必问、--auto 才不问"
- pm-refine 反例黑名单第 3 条从"需要 PM 逐个确认才能继续 → 反模式"改为"追问模式下不逐维确认才反模式；自主模式下逐维确认才反模式"

## Alternatives Considered

### A. 新建 pm-grill skill，pm-refine 不变

否决。理由：
1. 精炼的 8 维骨架是单一领域动作，拆成两个 skill 破坏语义完整性
2. grill-with-docs 是独立 skill 因为它不依附于既有动作；pm-refine 已有精炼动作，追问只是其执行方式
3. pm-need 编排器要加一个子步骤，比改 refine 内核更脏
4. --auto 链路要绕过 pm-grill，编排复杂度上升

### B. pm-refine 加 `--discuss` flag，默认仍自主推断

否决。理由：用户明确要求"正常模式默认追问，--auto 后全自动化"。默认自主推断意味着 PM 每次都要记得加 flag 才能讨论——违背"PM 自己也模糊时需要被引导"的初衷。

### C. 保留 pm-refine 不变，新建独立 pm-discover 做需求发现讨论

否决。理由：与 A 同源——pm-discover 会与 pm-refine 的 8 维重叠，造成"先 discover 讨论 8 维，再 refine 推断同样 8 维"的冗余。

## Consequences

**正向**：
- PM 模糊场景被覆盖——被引导问对问题比事后看信息缺口清单更有价值
- 与 grill-with-docs 的成熟追问机制对齐（一次一问、给推荐答案、查代码优先）
- --auto 链路完全不受影响，零确认体验保留

**负向**：
- pm-need 正常模式从"几秒出 PMContext"变为"可能十几分钟讨论"——PM 需有心理预期
- 现有 evals/pm-refine.json 的 3 个场景需重写——当前都假设自主推断
- 反例黑名单需反转，SKILL.md 的 Purpose/Context 段需重写
- 文档（README/CONTEXT）中"无需 PM 逐个确认"的措辞需全局修订

**已决定**（grilling 会话第 3-6 轮锁定）：
- **退出机制 = C+A 组合**：PM 可说"停"（已问维度落盘，未问维度强标 `[待确认]`）或"剩下的你自主"（降级为自主推断，未问维度走 `[假设]` 路径，置信度更高）。Agent 检测连续 3 个"不知道"也触发降级。
- **增量落盘纪律**：追问模式**不套用原子叠加**——每 2-4 个问题或每完成一个精炼维度（取先到者）增量落盘 PMContext 已回填项。理由：追问是多轮对话，可能被中断/超时/上下文丢失，增量落盘避免信息丢失。这与 ADR 0008 的"原子叠加"纪律冲突，但安全优先于原子性——追问模式的对话属性使得原子叠加风险过高。
- **推荐答案格式 = 三段式**：`推荐: <一句话> | 依据: <材料来源> | 备选: <1 个其它可能>`。备选不超过 1 个。
- **PM 回答处理**：采信 PM 的话标事实，来源记 `← 对话: PM 确认`；不追问 PM 给出的 X 的依据（PM 是领域权威）；若 X 与现有材料矛盾走 `[冲突]` 路径而非反问。
- **自主模式内部 loop 不落盘对话**：内部自问自答是隐式推理（对齐 CONTEXT.md 心智链不落盘纪律），结论走现有 `.loop/refine-step2/3/4.md` 路径，**不新增** `.loop/refine-discuss-*.md`。审计三元组已提供可追溯性，无需对话原文。理由：自主模式价值是 PM 零介入，事后审计对话违背 --auto 契约。
- **审计门优化（非精简）**：追问模式下审计门不再重复 PM 刚答过的 8 维细节，转而聚焦全局元信息（置信度分布 + 信息缺口 + 项目扫描发现 + 下一步选项）。**信息量不减，视角改变**——从"逐维细节确认"转为"全局全貌决策"。与自主模式对称：自主模式审计门展示完整置信度（PM 未参与需看全貌），追问模式审计门优化呈现（PM 已参与只需看全局元信息）。

**所有待定项已锁定，无遗留。**
