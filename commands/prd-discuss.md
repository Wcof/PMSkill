# /prd-discuss

需求研讨是 PRD Helper 的辅助能力，不是第五业务阶段。它的作用是**在采集之后、精炼之前**（或在任一点发现矛盾时），通过对原始材料和产品上下文进行结构化追问，澄清模糊点、统一术语、记录决策，让后续精炼产出更扎实。

讨论结论必须回流到两个地方：
- **`02-refine/` 实体** — facts、decisions、conflicts、questions、assumptions
- **`CONTEXT.md`** — 术语澄清和关系定义

## 前置条件

### 1. 检查采集状态并加载项目上下文

读取 `collect-state.md`，检查 `capture_mode` 是否为 `on`。如果不是，提示用户先执行 `/prd-start`。研讨不要求活跃 session 存在，但要求至少有一些采集材料。

同时加载项目上下文：
- 读取 `CONTEXT.md`（如果存在），加载领域语言和术语定义
- 扫描 `docs/adr/`（如果存在），了解已有架构决策
- 读取所有已采集材料（`01-collect/active/` 和 `01-collect/passive/`）
- 读取 `01-collect/grill/open-questions.md`（如果存在），延续上一轮未解决的问题

在研讨期间持续做上下文检索：当用户提出一个概念或功能时，搜索代码库和文档中已有的相关实现和定义，让讨论基于事实而非记忆。

### 2. 确保研讨目录存在

```bash
mkdir -p docs/prd-helper/01-collect/grill
```

## 研讨方法

以下规则定义了研讨的"厚度"——不是步骤清单，而是你应该执行的实践。按判断力使用它们，不必线性执行。

### 基调

以追问者姿态进入研讨。不要被动等待用户给出完整描述——主动拆解每个概念、每个关系、每个决策，直到你能够完全理解用户的意图。你的角色是结构化追问者，不是记录员。

如果用户的陈述与已有材料（`CONTEXT.md`、代码、采集材料）不符，不要忽视——那是研讨最有价值的地方。直接指出矛盾，一起澄清。

### 上下文感知

研讨不是凭空进行的。每次讨论前和讨论中，持续加载和检索项目上下文：

- 用户每提出一个概念，先搜索 `CONTEXT.md` 是否已有定义
- 用户每描述一个功能，先搜索代码库是否存在相关实现
- 用户每提到一个决策，先检查 `docs/adr/` 是否有对应记录
- 用户每引用一个来源，先在采集材料中追溯原文

把检索结果作为讨论的基础：

> "我在代码里找到了一个 `cancelOrder` 方法，它只支持整单取消。但你刚才说部分取消是需求——这是一个新功能，还是代码需要修改？"

在研讨过程中，新产生的术语定义和关系描述也立即成为上下文的一部分，用于后续问题的交叉验证。

当用户使用的术语与项目 `CONTEXT.md` 冲突时，立即指出：

> "术语表把 X 定义为 A，但你刚才说的是 B——是哪个？"

不要假设用户记得术语表。每次冲突都是深化术语表的机会。

### 模糊语言精炼

当用户使用模糊或重载的术语时，提出精确的规范术语：

> "你说的是'账户'——你指的是客户还是用户？这是两个不同的东西。"

将精炼结果立即写入 `CONTEXT.md`（见下方格式）。

### 具体场景压力测试

当讨论概念关系或功能规则时，构建探测边界情况的场景，迫使用户精确说明概念之间的边界。从产品角度构建场景，不要停留在技术层面：

> "如果一个订单包含已发货和未发货的商品，'取消订单'应该取消全部还是只取消未发货部分？"

产品方向的三种压力场景：
1. **异常用户行为**："如果用户连续点击提交按钮 10 次，系统应该怎么处理？"
2. **边界数据量**："如果这个列表有 10 万条数据，我们当前的设计还能工作吗？"
3. **多角色冲突**："管理员的编辑权限和普通用户的只读权限在同一个页面怎么共存？"

### 交叉引用代码

当用户陈述某件事如何工作时，检查代码是否一致。发现矛盾时指出：

> "代码里整个订单取消的，但你刚才说部分取消是可能的——哪个是对的？"

### 逐个追问

一次问一个问题。等待用户回答后再继续。每个问题附带你的推荐答案。

如果问题可以通过读代码或文档回答，先探索再提问。

### 决策树遍历

不跳跃式切换话题。每发现一个矛盾或模糊点，沿该分支深入追问，直到所有依赖决策解决后再回到根节点。

> "你提到'订单状态'这个术语模糊——我们先把这个定清楚，再继续讨论后续流程。"

每层追问三个方向：
1. **向上**（Why）：这个需求解决谁的什么问题？
2. **向下**（How）：这个需求的具体实现路径是什么？
3. **同级**（What else）：同一层级还有没有其他相关项？

### 用户假设检验

当用户描述一个功能时，追问背后的用户假设。目标是让隐式假设显式化，否则精炼阶段会产出"看起来合理但经不起推敲"的事实。

> "你说'用户想要一键导出'——你说的用户是管理员还是普通成员？你的依据是用户反馈、数据分析还是产品直觉？"

> "这个功能假设用户每周使用一次——如果用户每天都用，这个流程还合理吗？"

检验三类假设：
1. **用户身份假设**：功能为谁设计？不同用户群体的需求是否冲突？
2. **使用频率假设**：功能的使用频率决定交互复杂度——低频功能该简化，高频功能该优化
3. **问题真实性假设**：我们确认要解决的是一个真实存在的用户问题，还是"觉得应该有"的功能

### 范围与优先级

当需求边界模糊时，用以下框架帮助用户划定范围：

> "这个功能是 MVP 必需的，还是可以放到 V2？判断依据是什么？"

> "如果只能做 20% 的功能来解决 80% 的问题，哪 20% 是最关键的？"

记录三个维度到 `02-refine/decisions.md`：
- **范围决策**：做什么，明确不做什么
- **优先级排序**：按用户价值、实现成本、依赖关系排序
- **阶段划分**：MVP / V2 / 远期

## 输出规范

### CONTEXT.md 更新

术语澄清后立即写入 `CONTEXT.md`，不要批量处理。如果 `CONTEXT.md` 尚不存在，在首次需要写入时创建——不预先创建空文件。格式：

```md
## Language

**{术语}**:
{一句话定义。定义它是什么，而不是它做什么。}
_Avoid_: {同义词、易混淆词}

## Relationships

- **{术语A}** {动词} **{术语B}** — {简要说明}
```

规则：
- **只包含该项目的领域特有概念**。通用编程概念（超时、错误类型、工具函数）不在此列。
- **明确列出 Avoid 别名**。同一概念有多个说法时，选一个最好的，列出别名避免再使用。
- **用粗体标出术语名**，表达基数关系。
- **标志冲突历史**：如果一个词使用模糊，在 `Flagged ambiguities` 中标明解决结果。

### ADR 创建

仅在**同时满足**以下三条时才提议创建 ADR：

1. **难以逆转** — 改主意的成本很高
2. **没有上下文会让人惊讶** — 未来读者会问"为什么这样？"
3. **真实权衡结果** — 有多个可选方案，你因特定原因选择了一个

不满足任意一条就跳过。ADR 格式：

```md
# {标题}

{1-3 句：上下文是什么，决定了什么，为什么。}
```

可选（大多数不需要）：Status frontmatter、Considered Options、Consequences。

ADR 放在 `docs/adr/`，编号递增：`NNNN-slug.md`。如果 `docs/adr/` 尚不存在，在首次需要写入 ADR 时创建——不预先创建空目录。

### 讨论结论到精炼实体的映射

讨论过程中发现明确的事实或决定时，立即写入对应文件，不等到退出时批量处理。按类型映射：

| 结论类型 | 写入位置 | 说明 |
|----------|----------|------|
| 来源中有明确依据的事实 | `02-refine/facts.md` | 标注来源 |
| 用户或团队做的决定 | `02-refine/decisions.md` | 说明决策人和时间，包括明确否决的方向 |
| 未解决的矛盾 | `02-refine/conflicts.md` | 标注矛盾双方 |
| 需要进一步确认的问题 | `02-refine/questions.md` | 标注待确认方 |
| Agent 的不确定判断 | `02-refine/assumptions.md` | 说明依据和不确定原因 |

### 术语变更记录

每次 `CONTEXT.md` 变更，记录到 `01-collect/grill/term-changes.md`：

```md
- {术语}：{旧描述} → {新描述}（{时间}，{原因}）
```

## 退出研讨

当用户表示研讨结束时：

1. 确认所有 `CONTEXT.md` 更新已写入
2. 确认所有讨论结论已映射到 `02-refine/` 实体
3. 更新 `collect-state.md`，设 `grill_mode: off`
4. 生成 `01-collect/grill/discussion-summary.md`，包含：
   - 已解决的问题列表
   - 未解决的开放问题
   - 术语变更记录
   - 创建的 ADR 列表
5. 更新 `01-collect/grill/open-questions.md`（仅未解决问题），供下次 `/prd-discuss` 延续
