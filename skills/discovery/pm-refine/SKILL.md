# /pm-refine

> 你是一位资深产品分析师。摆在你面前的是刚收集完的原始材料——会议纪要、用户反馈、代码扫描结果、竞品分析。你的任务不是全部确认，而是**自主推断**，把杂音变成结构化的 PMContext。

对已收集材料自主推断澄清，区分事实/假设/冲突/待确认，沉淀成 PMContext。无需 PM 逐个确认。

## Purpose

对已收集材料自主推断澄清，区分事实/假设/冲突/待确认，沉淀成 PMContext。推断优先于追问——置信度低于 8 一律标假设。

## Context

刚收集完的原始材料需要结构化。本 skill 自主推断 8 个维度，沉淀为 PMContext 单文件。

## Instructions

Agent 对每个维度**先尝试从已有材料推断**：
- 材料中有明确依据 → 写为**事实**，标注来源
- 材料不足以确定但可合理推断 → 写为**推断**，标 `[假设]` 并附置信度(1-10)
- 材料完全缺失 → 标 `[待确认]`，记入信息缺口清单
- 不同材料矛盾 → 标 `[冲突]`，Agent 选择更可信来源并标注理由

## PM 自检维度（8 维全覆盖）

Agent **必须**对每个维度尝试推断，不可跳过。**🔴 规则：若跳过任一维度，PMContext 信息缺口不完整，下游 PRD/草图质量将下降。**

按推断优先级排序（P0 必推断、P1 强烈建议、P2 增量补充）：

### P0 — 决定 PMContext 是否可用（必须先推断）
1. **用户场景**：谁在什么场景下用？达到什么目的？
2. **边界条件**：异常路径——如果 X 失败呢？如果用户同时做 Y 呢？
3. **冲突检测**：不同来源/陈述矛盾时以哪个为准？

### P1 — 决定 PMContext 质量上限
4. **优先级**：必须做 vs 最好有，MVP 边界在哪？推断时参考优先级框架辅助决策：
   - **ICE**：Impact × Confidence × Ease，适合快速打分。Impact = Opportunity Score × 用户数，Confidence(1-10) 是对 Impact 的把握，Ease(1-10) 是实现难度倒数。
   - **RICE**：在 ICE 基础上将 Impact 拆为 Reach × Impact，适合用户规模差异大的场景。(R × I × C) / E。
   - **Opportunity Score**（Dan Olsen）：Importance × (1 − Satisfaction)，归一化到 0-1。高重要性 + 低满意度 = 最佳机会。**优先排问题（机会），不是排方案。**
   - **Kano**：必备/期望/兴奋/无差异/反向——用于判断功能类型，必备优先、兴奋做差异化。
5. **术语澄清**：模糊术语给出 Agent 理解的定义，标 `[假设]`
6. **现状平替与摩擦力**：用户目前用什么土办法凑合？最痛苦的点是什么？

### P2 — 增量增强（材料充分时再推断）
7. **技术与资源约束**：延迟要求？Token 成本？硬件限制？
8. **价值验证度量**：上线后看哪个指标证明做对了？推断时参考度量设计准则：
   - 度量必须反映用户价值交付，不是虚荣指标（PV/DAU 不算，任务完成率/留存率才算）
   - 度量必须是可行动的——指标变了，团队能知道该做什么
   - 度量必须可归因——能拆到具体功能/用户群，不是黑箱
   - 优先选比率型指标（转化率/留存率），不选绝对值型（总用户数）
   - 每个度量配一个阈值：低于阈值 = 假设不成立，考虑下线或重做
   - 度量要有采集时机：上线后第几天看、看多久
   - 先选 1 个核心度量 + 2-3 个辅助度量，不要超过 5 个

**推断纪律**：P0 三维必须有结论（事实/[假设]/[待确认]/[冲突] 之一），P1/P2 若材料不足可标 `[待确认]` 但不可跳过。

## 失败模式

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| `docs/pm-context/collect/` 不存在或为空 | **🔴 STOP**：输出"未收集材料，先运行 `/pm-collect` 或 `/pm-need`" | 不阻塞，提示后退出 |
| 材料总量 < 3 条 | 🟡 WARNING 标注"材料过少，推断置信度将偏低" | 不阻塞，继续推断但置信度普遍 ≤ 5 |
| P0 三维中任一维材料完全缺失 | 标 `[待确认]` 并记入信息缺口清单 | 不阻塞，但 PMContext 顶部加 ⚠️ "P0 维度 N 项待确认" |
| 同一维度材料互相矛盾 | 标 `[冲突]`，Agent 选更可信来源并标注理由 | 不强行二选一，冲突项写入信息缺口供 PM 审计 |
| 8 维全部推断完成但 `[假设]` 占比 > 50% | PMContext 顶部加 🟡 警示"假设占比高，建议补充材料重跑" | 不阻塞，落盘后供 PM 审计 |
| 增量更新时新材料与旧 PMContext 冲突 | 新推断标 `[冲突]` 并附新旧两版来源 | 不覆盖旧事实，由 PM 在审计门决定 |
| 置信度评分主观漂移（同维度不同材料置信度差异 > 4） | 复核评分依据，标注"置信度离散度高" | 不阻塞，但信息缺口清单记录该维度 |
| PMContext 文件已存在且未指定增量 | **🔴 STOP**：输出"PMContext 已存在，是否覆盖？覆盖将丢失历史推断" | 用户选否则退出；选是则全新走完流程 |

## 关联增强

推断时主动建立事实与规则、页面、验收之间的关联，确保 PMContext 内部无孤立项。

## 产物

沉淀为 `docs/pm-context/pm-context.md`，单文件自包含，按业务领域组织 heading：

```markdown
# PMContext: <需求名>
## 概述
### 问题与目标
### 现状平替与摩擦力
### 价值验证度量
## <页面/功能名>（如：用户登录）
### 事实
### 规则
### 验收
## <页面/功能名>（如：支付流程）
### 事实
### 规则
### 验收
## 全局约束
## 决策日志
| 决策点 | 选项A | 选项B | 最终选择 | 理由 | 来源 |
|--------|------|------|---------|------|------|
## 假设清单与验证计划
| 假设 | 置信度(1-10) | 风险类型 | 验证方式 | 成功指标/阈值 | 验证时机 |
|------|------------|---------|---------|--------------|---------|
## 风险项 [待确认] [假设] [冲突]
## 信息缺口
- <维度>：<缺什么，建议 PM 补充什么>
```

## 产出示例

基于上述会员中心材料的 PMContext 摘要：

```
# PMContext: 会员中心
## 概述
### 问题与目标
- 事实: 会员续费流程需重新填信息（← 对话: 用户反馈）
- [假设: 用户流失与续费流程负相关，8/10]（推断自续费率下降数据）
- [待确认] 当前续费转化率基线数据（PM 需补充分析）
### 现状平替与摩擦力
- 用户手动记录到期日（← 用户反馈）
- 无自动提醒机制（← 项目扫描: 代码中无提醒模块）
```

## 📝 实战提示

- **P0 三维必须有结论**：用户场景/边界条件/冲突检测是底线，材料不足也须标 `[待确认]`
- **ICE/RICE 优先选一个**：用户规模差异大用 RICE，否则用 ICE 更简单
- **假设置信度 ≤ 5 必须进信息缺口**：不要留在正文里冒充事实
- **增量更新时读旧 PMContext 做 diff**：不要让新材料覆盖旧事实

## 标记规则

- `[待确认]` — 材料完全缺失，Agent 无法推断
- `[假设]` — Agent 从已有材料推断，置信度 < 8
- `[冲突]` — 不同材料矛盾
- 每项内容要么行内标注来源（`← 来源: 项目扫描/知识库/对话/推断`），要么无来源。无来源的项自动标 `[假设]`。

## 增量更新

PMContext 是活文档。若 `pm-context.md` 已存在，Agent 读现有 PMContext + 新材料，只推断新增部分，增量写入。

## 不要做什么（反例黑名单）

| 反模式 | 为什么不要做 |
|--------|------------|
| Agent 跳过任一推断维度 | 8 维设计覆盖了 PM 最常遗漏的区域，跳过意味着信息缺口不完整 |
| 沉淀出孤立项不关联 | PMContext 中每项都应与其他项有引用关系，孤立项会降低 PMContext 作为"源"的价值 |
| 需要 PM 逐个确认才能继续 | 设计原则是 Agent 自主推断 + 审计门（PM 只审一次）|
| 置信度 7 的推断不标 `[假设]` | 置信度 8 以下按规则必须标 `[假设]`，否则 PM 会误认为事实 |
| 无来源的项不自动标 `[假设]` | 每项必须有来源。无来源=假设，这是追溯纪律的底线 |
| 忽略来源标注使下游无法追溯 | ai-prd 和 human-prd 都依赖 PMContext 中的来源标记做追溯 |
