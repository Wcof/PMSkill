# 双形态 PRD 同源骨架对比示例

> 本文件是 `/pm-prd` 的 Level 3 渐进披露资源。展示同一 PMContext 衍生出的 ai-prd 与 human-prd 如何同源不同写法。

## 输入：PMContext 片段（会员体系重构）

```markdown
## 会员权益
### 事实
- 当前仅月付会员，权益：广告去除 + 高清播放（← 知识库：会员规则.md）
- [假设: 用户希望有年付选项，7/10]（推断自 NPS 评分关联分析）
- [待确认] 年付用户 LTV 是否显著高于月付（PM 需补充财务数据）

### 规则
- 会员分三级：标准 / 高级 / 至尊
- [冲突] 权益是否包含线下活动——销售反馈包含，运营反馈不包含

### 验收
- 权益变更后 24 小时内全量同步至客户端
```

## AI PRD 形态（可执行契约）

```markdown
## US-1: 年付方案选择
As a 月付用户, I want to upgrade to 年付, so that 节省费用 ← PMContext: 会员权益/事实

### 验收标准
1. 月付用户可在"续费页"切换年/月付 tab
2. 年付价格必须 ≤ 月付 × 10
3. 切换后已付月费按比例抵扣 ← [假设: 财务支持比例退款，8/10]

## 实施规则
R-1: 会员三级（标准/高级/至尊），字段 `membership_level` 枚举 ← PMContext: 会员权益/规则
R-2: 权益变更后 24h 内全量同步客户端（定时任务 + 推送） ← PMContext: 会员权益/验收
R-3: ⚠️ 冲突项：权益是否含线下活动见风险项章节，暂不实现 ← PMContext: 会员权益/规则[冲突]

## 风险项
| 标记 | 内容 | 影响范围 | 处理方式 |
|------|------|---------|---------|
| [待确认] | 年付 LTV 数据 | R-1 定价 | 等待 PM 补充财务数据 |
| [冲突] | 线下活动权益 | R-3 | 需 PM 先解决冲突 |
```

## Human PRD �态（评审叙事）

```markdown
## 用户故事
🎯 月付用户 A，每次到期前没收到提醒，常忘记续费导致权益中断
- 业务价值：减少因忘记续费导致的流失，预计挽回 8% 续费率

## 实施规则
| 规则 | 为什么这样决定 |
|------|--------------|
| 会员分三级 | > 现有数据：85% 用户集中在基础权益，分级后可精准定价 |
| 到期前 7 天发 push 提醒 | > **为什么**：用户调研 80% 受访者选"提前一周" |

## 风险项
- [待确认] 年付 LTV 数据——若低于月付则年付定价需重估
- [冲突] 线下活动权益——销售与运营分歧，需 PM 在定价确认前敲定

## 追溯
- 年付方案 → PMContext: 会员权益/事实
- 三级会员 → PMContext: 会员权益/规则
- 权益同步时效 → PMContext: 会员权益/验收
```

## 同源对照要点

| 维度 | AI PRD | Human PRD |
|------|--------|-----------|
| 用户故事 | As-an 英文模板 + 追溯标记 | 中文自然叙事 + 业务价值 |
| 规则 | 可执行指令 + 字段名 | 决策表 + `> 为什么` 引用块 |
| 风险项 | 影响范围 + 处理方式 | 影响分析 + 如果错了怎么办 |
| 追溯 | 每条 `← PMContext: 章节` | 文末 `## 追溯` 小节汇总 |
| Agent Context | 必含（技术栈/目录/模块） | 不含（人类不需要） |

## 实战提示

- **`--auto` 跳过审计门**：适合原型快速迭代，正式交付前建议手动审计一轮
- **`--skip-human` 只出 AI 版**：内部技术评审时用，省去决策理由叙事
- **审计摘要必看未覆盖风险项**：这些是 PMContext 中 [待确认] 但 PRD 未处理的项
- **双形态必须同源**：ai-prd 和 human-prd 引用同一 PMContext，差异只在写法
- **冲突项不入用户故事**：两形态都把 [冲突] 项放在风险章节，不强行选定方向
