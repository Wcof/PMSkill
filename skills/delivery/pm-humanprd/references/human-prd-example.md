# 给人的 PRD 产出示例与评审友好叙事技巧

## 完整片段

假设 PMContext 已存在「会员体系重构」，给人的 PRD：

```markdown
# 会员体系重构 PRD

## 概述（Overview）
当前月付续费率连续 3 个季度下降（Q1: 68% → Q3: 56%），用户反馈"记不住到期日"和"想要年付但没选项"。

**为什么现在做**：竞品 A 和 B 均已推年付方案，窗口期有限；内部测算年付 LTV 是月付的 1.8 倍。

## 用户故事
🎯 **用户故事 1**：月付用户 A，每次到期前没收到提醒，手动记日历还常忘，导致服务中断后重新开通。
- 业务价值：减少因忘记续费导致的流失，预计续费率回升 10pp

🎯 **用户故事 2**：长期用户 B，愿意年付锁定价格，但当前只有月付选项。
- 业务价值：提升 LTV，年付用户留存率预计 > 90%

## 实施规则（Rules）
| 规则 | 为什么这样决定 | 审计摘要 |
|------|--------------|---------|
| 到期前 7 天发 push 提醒 | > 用户访谈 #3 提到"提前一周提醒最有效"，80% 受访者选 7 天 | 依据源2 + /pm-refine 8维之"优先级" |
| 年付价格 ≤ 月付 × 10 | > 竞品分析显示 10 倍是用户心理阈值，超 10 倍转化率骤降 | 依据源1 + /pm-refine 8维之"技术约束" |
| 升级差价按剩余天数比例退还 | > 财务要求"不可整月退款"，比例退还是合规最小代价 | 依据源2 + /pm-refine 8维之"边界条件" |

> **为什么选 7 天而非 3 天**：3 天太近用户来不及操作，14 天太早会被忽略。7 天是 NPS 调研最优解。

## 数据模型（Data）
- **MemberPlan**：会员订阅方案，关联 User，含类型（月/年）、起止日期、自动续费开关
- **PaymentRecord**：支付记录，关联 MemberPlan，含金额、方式、状态

## 验收标准（Acceptance）
- 月付用户可在"续费页"切换年/月付 tab
- 升级后立即生效，差价按比例退还到余额
- 到期前 7 天收到 push + 短信提醒

## 风险项（Risks）
- [待确认] 年付折扣比例 → 影响定价策略，需财务测算 LTV 后决定
- [假设] iOS push 送达率 > 80%（置信度 7/10）→ 若否，短信通道从备选升为必选
- [冲突] 产品说"到期当天也发提醒"vs 运营说"避免骚扰"→ 建议只发一次

## 超出范围（Out of Scope）
- 会员等级体系重构（本次只做付费周期扩展）→ 排除理由：等级体系涉及权益矩阵，独立迭代
- 多币种支持 → 排除理由：海外用户占比 < 5%，Phase 2 处理

## 决策日志摘要
| 决策点 | 选择 | 理由 |
|--------|------|------|
| 提醒时机 | 7 天前 | NPS 调研 80% 选 7 天 |
| 退还方式 | 按比例 | 财务合规要求 |
```

## 评审友好叙事技巧

| 技巧 | 说明 |
|------|------|
| 决策理由 > 结论 | 评审者要的是"为什么这么定"而非"定了什么" |
| 自然叙事 > As-an 模板 | 中文团队读"会员升级时差价按比例退还"比"As a member, I want..."高效 |
| 表格 ≤ 5 行 | 超过的拆为列表，人眼对长表格阅读效率低 |
| [待确认] 显式标注 | 评审者误以为已确认会导致决策偏差 |
| 每条规则配 > **为什么** | 引用块写决策理由，追溯 PMContext 事实 |
