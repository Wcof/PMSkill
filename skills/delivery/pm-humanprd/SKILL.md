# /pm-humanprd

> 你是一位资深产品经理，正在为产品评审会准备一份给人看的 PRD。评审者可能是 PM、业务方、设计师、技术负责人——他们的阅读习惯各不相同。**你的任务是让每个人都能快速看懂、有效反馈。**

从 PMContext 输出给人的 PRD，供人类阅读评审。注意：这是给**人**看的——评审者可能是 PM、业务方、设计师、技术负责人，阅读习惯和关注点各不相同。

## Purpose

从 PMContext 输出给人的 PRD，供人类阅读评审。给人看的 PRD 答的是"为什么这么定"而非"怎么做"——决策理由比结论更重要。

## Context

评审者可能是 PM、业务方、设计师、技术负责人，阅读习惯各不相同。本 skill 让每个人都能快速看懂、有效反馈。

## Instructions

读取 `docs/pm-context/pm-context.md`。若不存在 → 🔴 STOP：提示先运行 `/pm-need`。

- [ ] PMContext 已读取且非空
- [ ] 概述/页面定义/全局约束/假设清单/风险项/信息缺口全部提取
- [ ] 每条规则配决策理由（`> **为什么**` 引用块）
- [ ] 用户故事用自然叙事（非 As-an 英文模板）
- [ ] [待确认] 项显式标注不写成确定性要求
- [ ] 表格 ≤ 5 行，超过的拆为列表
- [ ] 输出生成摘要：用户故事数/规则数 + 格式纪律检查

## 产物

写入 `docs/pm-context/prd/human-prd.md`，与 ai-prd.md 同源同骨架，但写法针对人类读者调整。

结构模板：

```markdown
# <需求名> PRD

## 概述（Overview）
完整背景 + 决策理由。解释"为什么现在做这个"。

## 用户故事（User Stories）
自然语言描述，每一条加业务价值说明：
- 谁 + 在什么场景 + 要什么 + 为什么值（业务价值）
- 不要用 "As an... I want... so that..." 模板——用中文自然叙事

## 实施规则（Rules）
决策表 + "为什么这样决定"。每一条规则配：
- 规则描述（做什么）
- 决策理由（为什么，参考了哪些 PMContext 事实）
- 例外/边界（什么时候规则不适用）

## 数据模型（Data）
实体关系说明，用自然语言或简单表格。不要写字段类型——给人看不需要这个。

## 验收标准（Acceptance）
场景描述。每条验收标准配：
- 前置条件
- 操作步骤（简单版，不写成测试用例）
- 预期结果

## 风险项（Risks）
- `[待确认]` 项 + 影响分析（如果确认不了的影响）
- `[假设]` 项 + 置信度 + 如果错了怎么办
- `[冲突]` 项 + 决策方向

## 超出范围（Out of Scope）
列表 + 排除理由。写明"本次不做，因为……"。

## 决策日志摘要
从 PMContext 决策日志中提取关键决策，写给评审者看。
```

## 格式纪律

- 分段式标题（H2/H3），便于快速扫描
- 用户故事前加 🎯 标记
- 决策理由在 `> **为什么**` 引用块
- 表格 ≤5 行，超过的拆为列表

**🔴 CHECKPOINT** — 输出生成摘要：用户故事数/规则数 + [待确认]占比+警示 + 格式纪律检查。

## 不要做什么（反例黑名单）

| 反模式 | 为什么不要做 |
|--------|------------|
| 塞入 Agent Context 等机器可读内容 | 人类评审者不需要 |
| 用 As-an/I-want 英文模板写用户故事 | 中文团队评审，自然叙事更高效 |
| 把 `[待确认]` 写成确定性要求 | 评审者误以为已确认 |
| 表格超过 5 行 | 人眼对长表格阅读效率低于列表 |
| 省略决策理由只给结论 | 评审者不知道"为什么这么定" |

## 关联增强

生成时确保每条需求都追溯到 PMContext 中的具体项，无来源的需求标 `[待确认]`。
在 human-prd 文件末尾追加一个 `## 追溯` 小节，列出关键项在 PMContext 中的位置。

## 失败模式

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| `docs/pm-context/pm-context.md` 不存在 | **🔴 STOP**：输出"未找到 PMContext，先运行 `/pm-need <需求>`" | 不阻塞，提示后退出 |
| PMContext `[待确认]` 占比 > 40% | 输出 PRD 草案并在顶部加 🟡 警示横幅"含 N 项待确认，非正式 PRD" | 标记为 `human-prd.draft.md`，不覆盖已有正式 PRD |
| PMContext `[待确认]` 占比 > 70% | 输出大纲但顶部加 🔴 警示"信息严重不足，不宜评审" | 不生成正文，只输出信息缺口清单 |
| human-prd 与 ai-prd 同源项出现分歧 | 以 PMContext 为准修正 human-prd，标注修正项 `← 修正: PMContext` | 仍不一致则 git diff 记录分歧供 PM 审计 |
| 评审者非中文用户 | 检测 `$ARGUMENTS` 中的语言标记（`--en`/`--zh`/`-- bilingual`），输出对应语言版本 | 无标记则按 pm-setup 配置的语言偏好 |
| 决策日志为空（PMContext 无决策记录） | 在"决策日志摘要"章节标注"PMContext 未记录关键决策" | 不臆造决策理由，提示 PM 补充决策日志 |
| 表格超过 5 行关键信息 | 拆为列表 + 小表格，超过部分移入附录 | 不阻塞，但格式纪律要求拆分 |

## 产出示例

`/pm-prd --skip-ai 会员体系重构` → `prd/human-prd.md` 结构：

```
# 会员体系重构 PRD

## 概述（Overview）
当前月付续费率连续 3 个季度下降，用户反馈"记不住到期日"。
**为什么现在做**：Ofo 竞品已推年付，窗口期有限。

## 用户故事
🎯 用户故事 1：月付用户 A，每次到期前没收到提醒...
  - 业务价值：减少因忘记续费导致的流失

## 实施规则
| 规则 | 为什么这样决定 |
|------|--------------|
| 到期前 7 天发 push 提醒 | > 用户反馈中提到"提前一周提醒最有效" |
> **为什么**：用户调研 80% 受访者选"提前一周"

## 决策日志摘要
| 决策点 | 选择 | 理由 |
[关键决策...]
```

### 实战提示

- **覆盖评审者类型**：技术负责人关心规则和边界，业务方关心价值，设计师关心交互——各章节要有针对性
- **[待确认] 占比 > 30% 时不要当正式 PRD 发**：标记为草案，等确认后再正式化

| 反模式 | 为什么不要做 |
|--------|------------|
| 塞入 Agent Context、技术栈细节等机器可读内容 | 人类评审者不需要，反而造成干扰 |
| 用英文 As-an/I-want 模板写用户故事 | 中文团队评审，自然语言叙事更高效 |
| 把 `[待确认]` 写成确定性要求 | 评审者误以为已确认 |
| 表格超过 5 行关键信息 | 人眼对长表格的阅读效率低于列表 |
| 省略决策理由只给结论 | 评审者不知道"为什么这么定"，无法提有效反馈 |
